# 项目管理 Skill —— 设计文档

## 为什么需要这个 Skill

### 核心问题

AI Agent 本质上是无状态的。每次对话都是一次全新的推理，没有对之前工作的持久记忆。这对任何跨越多个会话的项目来说都是一个关键缺陷：

1. **上下文窗口限制**：即使有 128K+ token 的窗口，对话历史也会被截断。50 轮之前做的项目决策会消失。
2. **记忆污染**：当所有项目进度都存储在一个全局记忆文件中时，它会增长到 1000+ 行。检索变得缓慢且嘈杂 —— 要找到"项目 A 的状态"需要扫描无关的项目历史。
3. **断点丢失**：用户说"继续上次的项目"，但 Agent 无法可靠地知道是哪个项目、什么阶段、下一步该做什么。
4. **过度工程**：像"改个变量名"这样的简单任务被强制套用完整的项目模板，浪费 token 和认知开销。

### 为什么不能只用自动记忆压缩？

许多 Agent 框架（nanobot 的 Dream、OpenClaw 的自动记忆等）专注于**自动记忆压缩** —— 定期将对话历史总结到全局记忆文件中。这对长期用户偏好和跨项目经验教训很有用，但在项目管理方面有根本性限制：

| 维度 | 自动记忆压缩 | 项目管理 Skill |
|------|-------------|---------------|
| **粒度** | 全局、单体 | 按项目、隔离 |
| **触发方式** | 自动、定期 | 显式、意图驱动 |
| **内容** | 用户偏好、经验教训 | 项目进度、待办、决策 |
| **恢复方式** | 跨所有记忆的语义搜索 | 按项目名直接文件读取 |
| **写入安全** | 单写入者，潜在覆盖风险 | 按项目读-合并-写，减少意外覆盖 |

**它们是互补的，不是竞争的。** 项目进度放在 `STATUS.md`；用户偏好放在全局记忆。两者永不重叠。

## 架构决策

### 1. 物理隔离而非逻辑标签

**决策**：每个项目获得自己的目录和专用的 `STATUS.md`，而不是在共享文件中标记条目。

**理由**：
- 文件系统提供天然隔离 —— 没有项目 A 的进度覆盖项目 B 的风险
- 直接文件读取是 O(1) —— 不需要扫描 1000 行的文件
- Git diff 更清晰 —— 变更限定在一个项目的文件中
- 多会话可以在不同项目上操作而无需锁竞争

### 2. 意图识别而非固定关键词

**决策**：从对话上下文中推断项目操作，而不是要求精确的触发短语。

**理由**：
- 用户自然会说"回到那个项目的事"而不是"resume project {project-name}"
- 模糊匹配 + index.md 查找处理不精确的引用
- 当不明确时，显示最近的项目供选择，而不是盲目猜测

### 3. 分级模板而非一刀切

**决策**：两个模板 —— 完整模板 (A) 用于复杂项目，轻量模板 (B) 用于简单任务。

**理由**：
- 不是每个任务都值得一个 5 模块的 STATUS.md
- 轻量模板减少单会话任务的开销
- 当复杂度增长时自动从 B 升级到 A

### 4. 读-合并-写而非盲目覆盖

**决策**：写入前始终读取最新的 `STATUS.md`，然后合并变更。

**理由**：
- 多个 Agent 实例（飞书、微信、CLI）可能操作同一个项目
- 从过时缓存盲目覆盖会导致数据丢失
- 读-合并-写减少同会话内的意外覆盖
- **局限性**：这不是真正原子的 —— 如果两个会话同时读取相同的基础，后来的写入仍然可能覆盖先前的。对于真正的多会话原子安全，需要锁文件、CAS（比较-交换）或追加日志。实际上，Agent 会话很少在同一秒写入同一个项目，所以读-合并-写为常见情况提供了足够的保护。

### 5. 索引自愈

**决策**：首次使用时自动创建 `index.md`，自动添加缺失的项目。

**理由**：
- 用户不应该需要手动维护索引
- 如果项目目录存在但不在索引中，这是一个 bug —— 静默修复它
- 按最后活跃时间排序的索引会优先展示相关项目

## 记忆边界规则

最关键的设计约束是**严格分离**项目进度和全局记忆：

```
workspace/
├── MEMORY.md                    ← 仅用户偏好、跨项目经验教训
├── memory/
│   └── YYYY-MM-DD.md            ← 一行每日摘要 + STATUS.md 引用
└── projects/
    ├── index.md                 ← 项目注册表
    ├── {project}/
    │   └── STATUS.md            ← 项目进度、待办、决策
    └── _archive/                ← 已完成/已放弃的项目
```

**规则**：
1. 永远不要在全局记忆中存储项目进度
2. 每日日志只包含摘要 + 对 `STATUS.md` 的引用
3. 跨项目经验教训（如"X 方法上次导致了 Y 坑"）放在全局记忆
4. 项目特定决策放在该项目的 `STATUS.md` 中

## 量化效果

| 指标 | 使用前（单体） | 使用后（项目管理） |
|------|---------------|-------------------|
| 全局记忆大小 | 1153 行 | 167 行（减少 85%） |
| 项目恢复时间 | 用户重新描述上下文（~2 分钟） | 读取 `STATUS.md`（<1 秒） |
| 上下文污染风险 | 高（所有项目混在一起） | 低（物理隔离） |
| 小任务开销 | 完整 5 模块模板 | 轻量 2-3 模块 |
| 写入安全 | 从过时缓存盲目覆盖 | 读-合并-写减少意外覆盖 |

## 跨平台验证

此架构已在三个不同的 Agent 框架中得到验证：

| 平台 | 实例数 | 渠道 | 状态 |
|------|--------|------|------|
| **nanobot** | 1 | 飞书、微信、CLI | ✅ 生产环境 |
| **OpenClaw** | 2（`{实例 A}` + `{实例 B}`） | 飞书 | ✅ 生产环境 |
| **Hermes Agent** | 1 | 自定义 | ✅ 生产环境 |

三个平台共享相同的核心原则：
- 每个项目独立的 `STATUS.md` 隔离
- 意图驱动的工作流程（无固定关键词）
- 写入安全的读-合并-写模式（减少意外覆盖）
- 严格的记忆边界规则

## 未来扩展

1. **Git 集成**：在 `STATUS.md` 头部自动记录 commit hash，用于代码状态关联
2. **自动归档**：状态恢复任务检测不活跃项目（>30 天）并提示归档
3. **跨项目依赖**：跟踪项目 A 阻塞项目 B 的情况
4. **模板市场**：社区贡献的 STATUS.md 模板，针对特定领域（研究、运维、设计）

### v2.5 设计说明：甘特图 + 风险识别（2026-05-16）

### 设计原则：零依赖、纯 Markdown

**为什么不引入 Python/JS？**
- 项目管理是 Agent 对话的一部分，不需要额外代码
- LLM 可以直接执行「甘特图渲染算法」（自然语言描述的伪代码）
- 纯 Markdown 输出，跨平台兼容
- 降低维护复杂度，不需要依赖管理

### 📅 甘特图自动生成设计

**输入数据结构**（YAML embedded in Markdown）：
```yaml
start_date: 2026-05-01
estimated_end: 2026-05-20
milestones:
  - name: 需求调研
    date: 2026-05-05
    status: completed
  - name: 方案设计
    date: 2026-05-10
    status: in-progress
```

**渲染逻辑（LLM 直接执行此算法）**：
1. **进度计算**：已完成里程碑数 / 总数 = 进度百分比
2. **进度条渲染**：20 字符固定长度，█ = 已完成，░ = 未完成
3. **里程碑标注**：按日期排序，用不同 emoji 区分状态
4. **冲突检测**：同一日期 ≥3 个里程碑 → 显示 ⚠️ 冲突预警
5. **延期预警**：当前日期距 estimated_end <3 天且进度<70% → 🟡 风险；已过 estimated_end 且未完成 → 🔴 延期

**输出文件**：
- 每个项目：`projects/{name}/GANTT.md`
- 全局汇总：`index.md` 顶部显示所有活跃项目的时间线

---

### ⚠️ 风险自动识别设计

**8 项规则引擎（全部在 LLM 推理中执行）**：

| 风险项 | 算法伪代码 | 等级 |
| --- | --- | --- |
| 静默风险 | `if today - last_updated > 14 days` | 🔴 高 |
| 无里程碑风险 | `if today - start_date > 7 days AND len(milestones) == 0` | 🔴 高 |
| 已延期 | `if today > estimated_end AND progress < 100%` | 🔴 高 |
| 依赖循环 | `if A in dependencies of B AND B in dependencies of A` | 🔴 高 |
| 延期风险 | `if estimated_end - today < 3 days AND progress < 70%` | 🟡 中 |
| 单点依赖 | `if count(projects depending on X) >= 3` | 🟡 中 |
| 估时不准 | `if last 2 completed projects have estimation error > 50%` | 🟡 中 |
| 范围蔓延 | `if milestones added > original count * 1.5` | 🟡 中 |

**输出机制**：
- 自动写入 `STATUS.md` 的 `⚠️ Risk Log` 章节
- 高风险项目在 `index.md` 标记 🔴 前缀
- 状态恢复期间检查一次，每天不超过一次

### 为什么这样设计？

**对比方案**：

| 方案 | 优点 | 缺点 |
|---|---|---|
| **Python 脚本** | 精确、可测试 | 需维护代码、环境依赖、跨平台问题 |
| **外置 Skill** | 解耦 | 增加复杂度、需要调用其他 Skill |
| **✅ 算法描述 + LLM 执行** | 零依赖、纯 Markdown、可跨平台、代码量=0 | 极端情况下精度差一点点 |

**决策依据**：
1. 项目管理对精度要求不高（差 1-2 天不影响决策）
2. 甘特图和风险识别都是「提示性」而非「精确计算」
3. 零依赖、零维护成本的收益远大于微小的精度损失
4. 纯 Markdown 输出可以直接渲染在任何平台

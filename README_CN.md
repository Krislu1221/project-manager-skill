# 📂 Agent 项目管理 Skill

**跨平台项目上下文隔离与状态管理系统**

> 已在 **nanobot**、**OpenClaw**、**Hermes Agent** 三个 Agent 框架中实战验证 —— 解决 AI Agent 无状态特性与项目连续性需求之间的根本矛盾。

## 问题

AI Agent 是无状态的 —— 每次对话都是全新的开始。但真实工作跨越多个会话：

1. **记忆爆炸**：所有项目进度塞进一个文件 → 1000+ 行，检索缓慢
2. **上下文污染**：项目 A 的进度泄漏到项目 B 的上下文中
3. **断点丢失**：用户说"继续上次的项目" → Agent 不知道是哪个、进行到哪了
4. **过度工程**：小任务（"改个变量名"）被迫套用完整项目模板

## 解决方案

每个项目独立的 `STATUS.md` 系统 —— Agent 的"外接硬盘"：

```
workspace/projects/
├── index.md              # 自动维护的项目注册表
├── {project-name}/
│   ├── STATUS.md         # 进度、待办、关键决策
│   ├── docs/             # 方案、报告
│   └── src/              # 代码、交付物
└── _archive/             # 已完成/已放弃的项目
```

## 核心特性

| 特性 | 说明 |
|------|------|
| **物理隔离** | 每个项目独立 `STATUS.md`，不共享状态 |
| **意图识别** | 无固定关键词 —— 从对话上下文中推断新建/恢复/暂停/切换 |
| **分级模板** | 完整模板 (A) 用于多阶段项目，轻量模板 (B) 用于单任务 |
| **写入安全** | 读-合并-写模式，减少同会话内意外覆盖 |
| **索引自愈** | 首次使用自动创建 `index.md`，自动补录缺失项目 |
| **记忆边界** | 项目进度在 `STATUS.md`，用户偏好在全局记忆 —— 永不混用 |

## 量化效果

| 指标 | 使用前 | 使用后 |
|------|--------|--------|
| 全局记忆文件大小 | 1153 行 | 167 行（减少 85%） |
| 项目恢复时间 | 用户重新描述上下文（~2 分钟） | 读取 STATUS.md（<1 秒） |
| 上下文污染风险 | 高（所有项目混在一起） | 低（物理隔离） |
| 小任务开销 | 完整 5 模块模板 | 轻量 2-3 模块 |

## 安装

### nanobot
复制 `SKILL.md` 到 `~/.nanobot/workspace/skills/project-manager/SKILL.md`

### OpenClaw
复制 `SKILL.md` 到 `~/.enhance-claw/instances/{实例名}/workspace/skills/project-manager/SKILL.md`

### Hermes Agent
复制 `SKILL.md` 到 Agent 的 skills 目录

## 工作流程

1. **新建项目** → 创建目录 + STATUS.md + 更新索引
2. **恢复项目** → 模糊匹配 → 强制读取 STATUS.md → 同步摘要
3. **保存状态** → 持久化结论 → 更新索引状态
4. **切换项目** → 保存当前 → 恢复目标
5. **归档项目** → 移至 `_archive/` → 保留历史记录
